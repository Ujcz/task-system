# 12 状态推进规则

## 目标

把任务状态从“感觉差不多”改成“满足条件才推进”，避免代码完成但文档、验证、Review 或部署说明缺口仍然存在。

## 状态定义

| 状态 | 准入条件 | 退出条件 |
| --- | --- | --- |
| `analysis-ready` | 已读取 TAPD 或用户需求 | Context Pack 已完成 |
| `design-ready` | Context Pack 已完成 | Decision Pack 已完成，影响范围已确认 |
| `worktree-ready` | 涉及仓库和分支已确认 | worktree 与任务级 CodeGraph 已准备 |
| `code-ready` | 已授权实现 | 代码修改完成，自审前检查通过 |
| `review-ready` | 代码层面完成 | 自审完成，Review 输入已准备 |
| `test-ready` | Review 意见已处理 | 验证证据、测试说明、未验证项已准备 |
| `release-ready` | 提测说明完成 | 部署说明、回滚方案、观察点已准备 |
| `closed` | 发布或任务结束 | TAPD 回写、复盘沉淀完成 |

## 推进门禁

### 从分析到设计

必须有：

- TAPD 目标。
- 验收标准。
- 不涉及范围。
- 初步涉及仓库。
- 待确认问题。

### 从设计到实现

必须有：

- write_set。
- affected_surface。
- validation_plan。
- doc_plan。
- 用户授权实现。

### 从实现到 Review

必须有：

- 当前分支和 worktree 确认。
- 公共符号 impact 分析。
- 关键修改点自审。
- 未验证项记录。

### 从 Review 到提测

必须有：

- Review 意见处理状态。
- 修改后影响范围复核。
- Evidence Pack。
- 测试说明。
- H5/前端联调说明或明确不涉及。

### 从提测到发布

必须有：

- 部署说明。
- 回滚方案。
- 配置/SQL/Redis/MQ 变更说明。
- 发布后观察点。
- 残余风险。

## 禁止推进

出现以下情况时，不应进入下一状态：

- TAPD 目标不清楚。
- 不知道使用哪个 CodeGraph 索引。
- 不知道涉及哪些仓库。
- 公共契约未做影响分析。
- H5/前端可见接口没有联调文档。
- 数据库或配置变更没有部署和回滚说明。
- 未验证却声称“已通过”。
