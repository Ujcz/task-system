# 02 分析与设计

## 目标

在编码前完成代码结构理解、影响范围判断和方案设计，避免直接从需求跳到修改文件。

## 分析顺序

1. 使用 CodeGraph `context` 建立功能上下文。
2. 使用 `search` 定位核心类、方法、接口、DTO、DAO。
3. 使用 `callers` 查上游调用方。
4. 使用 `callees` 查下游依赖。
5. 使用 `impact` 评估修改影响。
6. 必要时使用 `trace` 追入口到目标方法的链路。
7. 补充读取配置、SQL、XML、YAML、pom、Redis key、MQ topic 等未完全由图谱覆盖的内容。

## 设计关注点

- 涉及仓库：`core`、`service`、具体 `kissu-server-*`。
- 涉及层次：Controller、Service、DAO、Redis、MQ、配置、定时任务。
- 是否涉及公共契约。
- 是否影响数据库表、字段、索引或 SQL。
- 是否影响缓存 key、TTL、失效策略。
- 是否影响 MQ 生产、消费或重试。
- 是否影响部署顺序、配置开关、回滚边界。


## 设计出口

满足以下条件才能进入实现：

- 修改点已明确。
- 影响范围已用 CodeGraph 或代码走查确认。
- 高风险点已有处理策略。
- 验证计划和交付文档计划已明确。
- 用户已明确授权进入实现。
