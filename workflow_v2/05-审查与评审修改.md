# 05 审查与 Review 修改

## 目标

把 Review 做成闭环：先自审，再处理外部意见，修改后重新确认影响。

## 自审清单

- 是否满足 TAPD 验收标准。
- 是否有非目标范围改动。
- 是否破坏兼容性。
- 是否遗漏调用方。
- 是否存在多层参数透传。
- 是否有配置、缓存、MQ、定时任务副作用。
- 是否需要补接口文档、测试说明或部署说明。
- 是否误改生成文件、临时文件、日志文件。

## Review 意见处理规则

1. 逐条读取 Review 意见。
2. 判断意见类型：缺陷、风格、可维护性、风险、疑问。
3. 合理意见直接修。
4. 不明确意见先解释或反问。
5. 技术上不成立的意见给出依据，不盲改。
6. 修改后重新做局部 impact 或调用关系确认。

## 输出：Review Response Pack

```text
Review Response Pack
- accepted:
- changed_files:
- rejected_or_deferred:
- reason:
- rechecked_impact:
- remaining_risks:
```

## 阶段出口

进入验证前至少满足：

- Review 意见均有处理状态。
- 修改后影响范围已重新确认。
- 残余风险已写明。
