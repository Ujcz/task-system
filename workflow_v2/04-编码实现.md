# 04 编码实现

## 目标

在已明确范围和影响的前提下，以最小、可审查、可验证的方式落地代码。

## 实现原则

- 优先沿用仓库已有模式。
- 不做无关重构。
- 不把多个主题混在一次改动中。
- 公共持久层优先落在 `Kissu-Server-Core`。
- 业务仓只调用公共层，不在业务模块里随意新增持久层三件套。
- 遵循“谁消费，谁计算；谁拥有，谁构建”。
- 不把 `dateList`、`redisKeySuffix`、缓存 key 片段等下层专用中间态多层透传。

## 分层规则

- Controller：参数接收、鉴权入口、简单编排。
- Service：业务语义、规则判断、调用组合。
- DAO / Mapper / Data：持久层访问，优先遵循 Core 既有模式。
- Schedule：定时任务入口、主从判断、异常兜底。
- Redis / MQ：按项目既有封装和 key/topic 规则接入。

## 高风险门禁

命中以下任一情况，必须先补充影响分析和验证计划：

- 公共 DTO、VO、BO、枚举、常量、工具类。
- proto 或生成类。
- 数据库表、字段、索引、SQL。
- Redis key、TTL、缓存兼容。
- MQ topic、消息体、消费者。
- 定时任务、异步队列、线程池。
- 权限、风控、支付、结算、活动策略。

## 实现出口

进入 Review 前至少满足：

- 修改范围与 Decision Pack 一致。
- 没有在主干目录误改。
- 关键公共符号已做 impact。
- 中文文件 UTF-8 正常。
- 已记录未验证项。
