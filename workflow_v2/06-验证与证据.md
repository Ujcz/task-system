# 06 验证与证据

## 目标

用明确证据描述完成度，不把“代码已改”误说成“任务完成”。

## 证据等级

| 等级 | 含义 |
| --- | --- |
| L0 | 仅代码走查和 CodeGraph 分析 |
| L1 | 局部命令、静态检查、定向脚本或影响分析 |
| L2 | 模块编译、定向测试或关键单元测试 |
| L3 | 联调、真实环境验证、提测通过或发布后观察 |

## 什么时候必须建议验证

- 公共契约变化。
- `pom.xml` 变化。
- proto 变化。
- 数据库持久层变化。
- 跨模块调用变化。
- Redis、MQ、定时任务、配置迁移。
- 用户明确要求“编译”“跑测试”“验证”“自测”。

## 输出：Evidence Pack

```text
Evidence Pack
- evidence_level:
- executed:
- passed:
- failed:
- not_executed:
- reason:
- code_walkthrough:
- codegraph_checks:
- residual_risks:
- recommended_next_validation:
```

## 表述规则

- 未跑编译就不能说“编译通过”。
- 未跑测试就不能说“测试通过”。
- 只能代码走查时，应写“代码层面判断”。
- 有未验证项时，必须写原因和建议补验路径。
