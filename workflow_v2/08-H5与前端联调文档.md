# 08 H5 与前端联调文档

## 目标

当后端新增或修改任何前端可见接口时，单独产出面向 H5、客户端、Admin 前端或 Web 前端的联调文档。它的目标读者是接入方，不是后端实现者。

## 什么时候必须产出

命中以下任一情况，就需要准备联调文档：

- 新增接口。
- 修改接口路径、请求方法、请求参数或响应字段。
- 修改错误码、错误文案或异常语义。
- 修改鉴权、权限、灰度、开关或公共参数。
- 修改分页、排序、过滤、状态枚举。
- 修改 H5、客户端、Admin 前端可见的数据展示结果。
- 返回字段存在协议字段名和真实 HTTP JSON 字段名不一致。

## 必须区分的接入对象

联调文档要明确写清楚本次涉及哪类接入方：

- H5 接口。
- 客户端接口。
- Admin 前端接口。
- Web 前端接口。
- 内部诊断或测试工具接口。

如果只涉及其中一类，要明确写“不涉及其他接入对象”。例如：

```text
本次仅涉及 Admin 前端接口，不涉及 H5 和客户端接口。
```

## 文档内容

```text
H5 / 前端联调说明
- 接入对象:
- 需求背景:
- 接口清单:
- 环境与域名:
- 鉴权与公共参数:
- 请求方法和路径:
- 请求参数:
- 响应字段:
- 真实 HTTP JSON 示例:
- 错误码和异常文案:
- 前端展示或交互建议:
- 兼容性说明:
- 不涉及接口:
- 联调验收场景:
- 后端已验证项:
- 后端未验证项:
```

## 路径和配置

接口路径不能直接照抄 Spring 注解里的占位符。

如果代码里是：

```java
@RequestMapping("${server.baseurl}liveRank")
```

必须读取 `application*.yml` 或 `properties` 中的 `server.baseurl`，再写成实际路径。无法确认环境配置时，要标注：

```text
实际路径需按目标环境的 server.baseurl 配置确认。
```

## 字段规则

接口文档必须区分三类字段：

- Java DTO / VO 字段名。
- 协议字段名。
- 真实 HTTP JSON 返回字段名。

如果三者一致，也要写明“一致”。如果不一致，要写映射关系。

示例：

| 含义 | Java 字段 | HTTP JSON 字段 | 说明 |
| --- | --- | --- | --- |
| 用户 ID | `userId` | `user_id` | 前端以 `user_id` 为准 |

## 响应示例

必须至少提供一份真实 HTTP JSON 口径的响应示例。不要只给 Java 字段名或 protobuf 字段名。

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "user_id": 123,
    "status": 1
  }
}
```

## 错误码和异常

需要写清楚：

- 哪些错误前端需要展示。
- 哪些错误只需要兜底提示。
- 是否使用英文 runtime msg。
- 是否存在参数错误、权限错误、状态冲突、配置缺失。

## 联调验收场景

至少覆盖：

- 正常请求。
- 空数据。
- 分页边界。
- 参数非法。
- 权限不足。
- 配置关闭或未命中。
- 老版本兼容。
- 多端差异：H5、客户端、Admin 前端如有差异需分别列出。

## 与测试文档的区别

前端联调文档回答“前端怎么接”。测试文档回答“测试怎么验”。两者可以互相引用，但不要混成一份。
