# 个人任务系统（AI + Java 后端）

用 Git + Markdown + AI 构建的**个人级任务管理系统**  
不依赖复杂工具，不引入团队协作成本  
只解决一个问题：**让一个后端开发者，把事想清楚、做干净、交得出**

---

## 一、背景（为什么会做这个系统）

作为一名 Java 后端开发工程师，我的日常工作已经从：
- 单一功能开发
- 线性任务推进
- 只关注代码实现

变成了：
- 多任务并行
- 接口 / 数据 / 配置多方协作
- 代码 + 文档 + 变更说明一起交付

在 AI 浪潮下，这种转变被进一步放大：
- AI 能写代码
- AI 能补文档
- 但我必须**知道自己在干什么、做到哪一步、交付了什么**

传统做法（脑子记、临时文档、零散任务）已经撑不住了。

---

## 二、遇到的问题（痛点）

在实际工作中，我反复遇到这些问题：

### 1️⃣ 任务状态不清楚
- 做了很多事，但说不清进度
- 并行任务互相干扰
- 经常“突然发现漏了东西”

### 2️⃣ 交付物不完整
- 只有代码
- 文档最后补，质量不高
- SQL、配置、影响面说明缺失

### 3️⃣ 任务管理成本高
- 工具复杂，维护成本比任务还大
- 不想为了管理任务，再学一套系统
- 团队协作工具对个人来说太重

### 4️⃣ AI 用不起来
- 不知道什么时候用 AI
- 给不出清晰上下文
- AI 输出不可控、不可复用

---

## 三、这个系统要解决什么

| 问题 | 解决方式 |
|---|---|
| 任务混乱 | 一个任务 = 一个 Markdown 文件 |
| 状态不清 | 状态 = 文件夹位置 |
| 并行失控 | 限制 doing 中任务数量 |
| 文档拖延 | 文档模板 + AI 生成 |
| 交付不稳 | 交付前检查清单 |
| AI 难用 | 固定 Prompt + 固定上下文 |

---

## 四、设计原则（为什么这么做）

### ✅ 极简
- 不引入数据库
- 不引入服务
- 不引入复杂概念

### ✅ 可迁移
- 换公司、换项目、换工具都能带走
- 纯文本 + Git

### ✅ 工程化
- 任务像代码一样管理
- 有版本、有历史、可回溯

### ✅ AI 友好
- 结构化输入
- 可重复使用的 Prompt
- 输出可直接进入交付物

## 五、目录结构说明

task-system/
├── backlog/          # 待办任务
├── doing/           # 进行中（建议 ≤ 3 个）
├── review/          # 待交付 / 待评审
├── done/            # 已完成任务
├── templates/       # 任务 & 文档模板
├── prompts/         # AI 提示词
├── assets/          # 物料（SQL、配置、图）
├── .gitignore
└── README.md

## 六、使用方式（单任务项目制）

每一个任务都是一个独立文件夹，包含该任务的所有交付物。

### 任务文件夹结构
任务名/
├── README.md     # 任务说明 & 进度
├── design.md     # 技术方案
├── api.md        # 接口文档（前端）
├── test.md       # 测试要点（测试）
├── product.md    # 功能说明（产品）
├── deploy.md     # 发布 & 回滚（自己）
└── assets/       # SQL / 配置 / 附件

### 状态流转
1-待办 → 2-进行中 → 3-待交付 → 4-已完成

只需移动文件夹，无需修改内容。

## 七、每天 10 分钟执行 SOP

### ☀️ 早上（3 分钟）
- 确认 doing 中的任务
- 必要时从 backlog 拉新任务

### 🌤 白天
- 专注执行，不折腾任务管理

### 🌙 晚上（7 分钟）
- 更新任务进度
- 勾选子任务
- 移动任务状态
- Git 提交

## 十一、小记功能（notes）

用于记录临时想法、关键点、待确认事项、排查过程等。

### 特点
- 不属于正式任务
- 不进入状态流转
- 只服务于你自己

### 使用方式
- 文件名：`YYYY-MM-DD-关键词.md`
- 内容不限，越短越好
- 不需要状态、不需要交付

### 常见用途
- 线上问题排查过程
- 临时想到的优化点
- 需要二次确认的技术细节
- 会议中不想忘的关键信息

### 生命周期
小记不是永久资产：
- 消化后删除
- 或合并进任务
- 或升级为 backlog 任务
