# 三方关联工作模式

本文说明后续如何用当前任务系统串联个人任务仓库、代码仓库和 TAPD，作为日常协作的固定工作方式。

## 一、系统分工

| 系统 | 位置 / 来源 | 职责 |
|---|---|---|
| 个人任务仓库 | `D:\jcz\task-system` | 记录任务状态、分析结论、方案、交付物和协作过程 |
| 代码仓库集合 | `D:\jcz\source` | 承载实际代码仓库、分支、提交、联调和验证结果 |
| TAPD | TAPD 需求 / 缺陷 / 任务 | 承载正式需求来源、评审意见、状态流转和对外协作文档 |

三者之间不依赖自动同步，默认通过任务文件中的“关联信息”保持可追踪关系。

## 二、主关联键

默认以 TAPD 单据作为主关联键。一个任务可以关联：

- 一个 TAPD 主需求。
- 一个或多个 TAPD 子需求、任务或缺陷。
- 一个或多个代码仓库。
- 一个统一任务分支。
- 一组本地 worktree。
- 若干 commit、PR、Wiki、发布记录。

如果任务暂时没有 TAPD，则先以本地任务名称作为临时关联键；后续拿到 TAPD 链接后再补齐。

## 三、任务文件建议记录字段

正式任务的 `README.md` 中建议维护以下区块：

```markdown
## 关联信息

| 类型 | 内容 |
|---|---|
| TAPD 项目 | workspace_id / 项目名 |
| TAPD 单据 | 类型 / ID / 链接 |
| 代码仓库 | Kissu-Server-Core, kissu-server-rank |
| 分支 | feature/module/change_summary |
| Worktree | D:\jcz\worktrees\feature-module-change_summary\... |
| Commit | TAPD 关联关键字 / 本地提交 |
| PR | 链接或待补充 |
| TAPD Wiki | 链接或待补充 |

## 当前状态

- 阶段：
- 已确认：
- 待确认：
- 风险：
```

## 四、默认协作流程

### 1. 接入需求

用户提供 TAPD 链接、ID、需求正文或口头描述后，先完成以下动作：

1. 识别 TAPD 项目、类型和 ID。
2. 读取需求正文、评论、父子关系和当前状态。
3. 提炼目标、验收标准、影响范围和待确认问题。
4. 判断涉及的代码仓库和模块。
5. 在任务仓库中建立或更新对应任务。

### 2. 任务拆解

一个正式任务应尽量补齐：

- 任务说明：为什么做、做什么、不做什么。
- 技术方案：模块、流程、数据、风险。
- 接口文档：请求、响应、错误码、前后端约束。
- 测试要点：正常、异常、边界、回归范围。
- 产品说明：用户可感知变化和影响面。
- 发布回滚：SQL、配置、灰度、验证、回滚方式。

### 3. 代码关联

涉及代码开发时，默认按以下约定执行：

- 代码仓库根目录：`D:\jcz\source`。
- 多仓任务使用同一个分支名。
- 新任务分支默认格式：`feature/{business-module}/{change-summary}`。
- 推荐 worktree 根目录：`D:\jcz\worktrees`。
- 主 checkout 保持在 `D:\jcz\source\<repo>`，功能开发优先在 worktree 中进行。

示例：

```text
分支：feature/rank/score_opt
任务目录：D:\jcz\worktrees\feature-rank-score_opt
仓库：
- D:\jcz\worktrees\feature-rank-score_opt\Kissu-Server-Core
- D:\jcz\worktrees\feature-rank-score_opt\kissu-server-rank
```

### 4. TAPD 回写

TAPD 默认作为正式协作出口，但写操作需要用户明确授权。

默认行为：

- 可以读取 TAPD 需求、任务、评论并总结。
- 可以生成 TAPD 评论、Wiki、交付说明草稿。
- 不主动覆盖 TAPD 正文。
- 不主动修改 TAPD 状态、工时、负责人或字段。
- 不主动写评论、更新 Wiki 或回写正文。

只有用户明确说“写回 TAPD”“更新 Wiki”“评论到需求下”“修改状态”等，才执行 TAPD 写操作。

## 五、常用输入方式

### 新建任务

```text
根据这个 TAPD 新建任务：
链接：...
关联代码仓库：core、kissu-server-rank
先分析，不改代码
```

### 推进开发

```text
这个任务可以开始实现。
代码仓库：core、kissu-server-rank
分支：feature/rank/score_opt
```

### 补齐交付物

```text
把这个任务推进到待交付，并补齐接口文档、测试要点、发布回滚说明。
```

### 回写 TAPD

```text
根据当前任务内容生成 TAPD 交付说明草稿，先不要写回。
```

或：

```text
把这份交付说明写回 TAPD Wiki，并在主需求评论区追加 Wiki 链接。
```

## 六、状态流转规则

任务仓库中的目录位置表示状态：

| 目录 | 含义 |
|---|---|
| `1.待办（源头：新需求、待分析的任务）` | 已记录，待分析或待排期 |
| `2.进行中（核心：当前正在做的事（建议≤3个））` | 正在分析、设计、开发或联调 |
| `3.待交付（出口：开发完，等测试验收）` | 开发完成，等待测试、验收或发布 |
| `4.已完成（归档：已上线已完结的历史记录）` | 已上线、已验收或已明确关闭 |
| `5.小记（缓冲区：临时灵感、排查记录、怕忘的点）` | 临时信息，不进入正式任务流转 |

移动任务目录前，应确认任务内容已反映当前真实状态。

## 七、默认边界

- 先分析、阅读、设计和评审，再进入修改。
- 未经明确授权，不修改代码仓库、任务仓库或 TAPD。
- 所有中文文档、配置、脚本和说明默认使用 UTF-8 编码，不主动添加 BOM。
- 没有验证证据时，不声称“已通过”“可上线”。
- 跨仓任务需要明确记录每个仓库的分支、提交和验证结果。
- 长文档默认沉淀在任务仓库或 TAPD Wiki，TAPD 主需求评论只放摘要和链接。

## 八、最终目标

让每个需求都能回答清楚：

1. 来源是什么：对应哪个 TAPD。
2. 做了什么：任务仓库中有完整说明。
3. 改了哪里：关联到具体代码仓库、分支和提交。
4. 怎么验证：有测试要点和验证结果。
5. 怎么交付：有接口、SQL、配置、发布和回滚说明。
6. 后续怎么看：TAPD、任务仓库和代码仓库之间可以互相追溯。
